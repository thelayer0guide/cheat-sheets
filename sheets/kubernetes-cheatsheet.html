Kontext & Cluster
Cluster-Info
kubectl cluster-info kubectl cluster-info dump # ausführlich kubectl version # Client + Server kubectl api-resources # alle Ressourcen kubectl api-versions # API-Gruppen
Kontexte (kubeconfig)
kubectl config get-contexts kubectl config current-context kubectl config use-context my-cluster kubectl config set-context --current \ --namespace=production kubectl config view # kubeconfig anzeigen
Nodes
kubectl get nodes kubectl get nodes -o wide # IP, OS, Kernel kubectl describe node node01 kubectl top node # CPU/RAM (metrics-server) kubectl cordon node01 # kein Scheduling kubectl drain node01 --ignore-daemonsets kubectl uncordon node01
💡 KUBECONFIG-Env-Variable überschreibt den Standardpfad ~/.kube/config. Mehrere Dateien: KUBECONFIG=a.yaml:b.yaml kubectl ...

Pods
Auflisten & Inspizieren
kubectl get pods kubectl get pods -o wide # Node, IP kubectl get pod mypod -o yaml # vollständiges Manifest kubectl describe pod mypod kubectl get pods -l app=nginx # Labels filtern kubectl get pods --field-selector \ status.phase=Running
Erstellen & Löschen
# Schnell: imperativer Pod kubectl run mypod --image=nginx:alpine # Aus Manifest kubectl apply -f pod.yaml kubectl delete -f pod.yaml kubectl delete pod mypod kubectl delete pod mypod --force
Logs & Exec
kubectl logs mypod kubectl logs mypod -f # follow kubectl logs mypod --tail=100 kubectl logs mypod -c sidecar # Container wählen kubectl logs mypod --previous # letzter Absturz # Shell im Pod kubectl exec -it mypod -- bash kubectl exec -it mypod -c sidecar -- sh
Port-Forward & Kopieren
# Lokalen Port fortwarden kubectl port-forward mypod 8080:80 kubectl port-forward svc/myservice 9090:9090 # Dateien kopieren kubectl cp mypod:/etc/config ./config kubectl cp ./local.conf mypod:/tmp/
💡 kubectl run --rm -it --image=busybox -- sh startet einen temporären Debug-Pod, der nach dem Beenden automatisch gelöscht wird.
📋
Deployments
Erstellen & Skalieren
# Imperativ kubectl create deployment web \ --image=nginx:1.25 --replicas=3 # Aus Manifest kubectl apply -f deployment.yaml # Skalieren kubectl scale deployment web --replicas=5 kubectl autoscale deployment web \ --min=2 --max=10 --cpu-percent=70
Updates & Rollbacks
# Image aktualisieren kubectl set image deployment/web \ nginx=nginx:1.26 # Rollout-Status kubectl rollout status deployment/web kubectl rollout history deployment/web # Rollback kubectl rollout undo deployment/web kubectl rollout undo deployment/web \ --to-revision=2
Deployment YAML (Kurzform)
apiVersion: apps/v1 kind: Deployment metadata: name: web spec: replicas: 3 selector: matchLabels: {app: web} template: metadata: labels: {app: web} spec: containers: - name: nginx image: nginx:1.25 ports: - containerPort: 80 resources: requests: {cpu: 100m, memory: 128Mi} limits: {cpu: 500m, memory: 256Mi}
💡 RollingUpdate (Standard): Null-Downtime-Updates. Recreate: alle alten Pods werden zuerst beendet — kurze Downtime.

Services & Networking
Service-Typen
# ClusterIP (intern, Standard) kubectl expose deployment web \ --port=80 --target-port=80 # NodePort (Host-Port) kubectl expose deployment web \ --type=NodePort --port=80 # LoadBalancer (Cloud) kubectl expose deployment web \ --type=LoadBalancer --port=80
Service anzeigen & löschen
kubectl get svc kubectl get svc -o wide kubectl describe svc web kubectl delete svc web
Ingress
kubectl get ingress kubectl describe ingress my-ingress kubectl apply -f ingress.yaml # Ingress YAML (nginx-ingress) apiVersion: networking.k8s.io/v1 kind: Ingress metadata: name: my-ingress annotations: nginx.ingress.kubernetes.io/rewrite-target: / spec: rules: - host: app.example.com http: paths: - path: / pathType: Prefix backend: service: name: web port: {number: 80}
💡 DNS-Name im Cluster: servicename.namespace.svc.cluster.local — Services sind innerhalb des Clusters automatisch per DNS erreichbar.

ConfigMaps & Secrets
ConfigMap erstellen
# Aus Literalwerten kubectl create configmap app-config \ --from-literal=DB_HOST=postgres \ --from-literal=DB_PORT=5432 # Aus Datei kubectl create configmap nginx-conf \ --from-file=nginx.conf kubectl get configmap app-config -o yaml
Secrets erstellen
# Generic Secret kubectl create secret generic db-secret \ --from-literal=password=s3cr3t # Docker Registry Secret kubectl create secret docker-registry regcred \ --docker-server=ghcr.io \ --docker-username=$USER \ --docker-password=$TOKEN kubectl get secret db-secret -o jsonpath=\ '{.data.password}' | base64 -d
Als Env / Volume nutzen
# Als Umgebungsvariable env: - name: DB_PASSWORD valueFrom: secretKeyRef: name: db-secret key: password # Als Volume volumes: - name: config configMap: name: app-config volumeMounts: - mountPath: /etc/config name: config
⚠️ Secrets sind nur Base64-kodiert, nicht verschlüsselt! Für echte Verschlüsselung: Sealed Secrets oder External Secrets Operator verwenden.

StatefulSet · DaemonSet · Job
StatefulSet
kubectl get statefulsets # sts kubectl scale sts mydb --replicas=3 kubectl rollout restart sts mydb # PVCs bleiben nach Pod-Löschung erhalten
DaemonSet
kubectl get daemonsets # ds kubectl describe ds fluentd -n kube-system kubectl rollout restart ds fluentd # Läuft auf jedem (passenden) Node
Job & CronJob
# Job einmalig ausführen kubectl create job pi-calc \ --image=perl -- perl -Mbignum=bpi -wle \ "print bpi(100)" kubectl get jobs kubectl get cronjobs # cj # CronJob pausieren kubectl patch cronjob my-cj -p \ '{"spec":{"suspend":true}}'
💡 StatefulSet: stabile Netzwerkidentität + geordnetes Hoch-/Runterfahren. Ideal für Datenbanken wie PostgreSQL, Cassandra, Kafka.

RBAC
ServiceAccount & Rollen
kubectl create serviceaccount my-sa kubectl get serviceaccounts # sa # ClusterRole (clusterweit) kubectl create clusterrole pod-reader \ --verb=get,list,watch \ --resource=pods # Namespace-spezifische Rolle kubectl create role pod-reader \ -n staging --verb=get,list \ --resource=pods
RoleBindings
kubectl create rolebinding read-pods \ --role=pod-reader \ --serviceaccount=staging:my-sa \ -n staging kubectl create clusterrolebinding admin-bind \ --clusterrole=cluster-admin \ --user=alice kubectl get rolebindings kubectl get clusterrolebindings
Berechtigungen prüfen
# Eigene Rechte kubectl auth can-i create pods kubectl auth can-i delete pods -n prod # Für anderen User kubectl auth can-i list nodes \ --as=alice kubectl auth can-i get secrets \ --as=system:serviceaccount:default:my-sa
⚠️ cluster-admin gibt vollen Zugriff — sparsam einsetzen! Principle of Least Privilege: nur nötige Verben und Ressourcen gewähren.
💾
Storage (PV / PVC)
Überblick
kubectl get pv # PersistentVolumes kubectl get pvc # PVC (Namespace) kubectl get storageclass # sc kubectl describe pvc my-pvc
PVC YAML
apiVersion: v1 kind: PersistentVolumeClaim metadata: name: my-pvc spec: accessModes: [ReadWriteOnce] storageClassName: standard resources: requests: storage: 10Gi
💡 ReadWriteMany (RWX): benötigt NFS oder Ceph/RBD. ReadWriteOnce (RWO): nur ein Node gleichzeitig — für die meisten Datenbanken ausreichend.

Debugging & Troubleshooting
Pod-Status prüfen
kubectl get pod mypod -o yaml kubectl describe pod mypod kubectl get events --sort-by='.lastTimestamp' kubectl get events -n default --field-selector \ involvedObject.name=mypod
Debug-Pods & Ephemeral Container
# Temporärer Debug-Pod kubectl run debug --rm -it \ --image=nicolaka/netshoot -- bash # Ephemeral Container (ab k8s 1.23) kubectl debug -it mypod \ --image=busybox --target=mycontainer # Kopie eines Pods mit anderem Image kubectl debug mypod -it \ --copy-to=debug-pod \ --set-image=*=ubuntu
Ressourcen & Status
kubectl top pods # Ressourcenverbrauch kubectl top pods --sort-by=memory kubectl top nodes # Alle nicht-laufenden Pods kubectl get pods -A --field-selector \ 'status.phase!=Running' # Pending Pods (Scheduling-Problem) kubectl get pods --field-selector=\ status.phase=Pending
Häufige Pod-Zustände
Pending # kein passender Node CrashLoopBackOff # Container crasht wiederholt ImagePullBackOff # Image nicht pull-bar OOMKilled # Speicherlimit überschritten Terminating # wird beendet (Finalizer?) Error # Exitcode != 0
💡 CrashLoopBackOff: kubectl logs --previous zeigt Logs des letzten Abbruchs. ImagePullBackOff: Registry-Zugriff und imagePullSecrets prüfen.

Output & Selektoren
Output-Formate
kubectl get pods -o yaml # vollständig kubectl get pods -o json # JSON kubectl get pods -o wide # erweiterte Tabelle kubectl get pods -o name # nur Namen # Benutzerdefinierte Tabelle kubectl get pods -o custom-columns=\ NAME:.metadata.name,STATUS:.status.phase
JSONPath & Filtern
# Pod-IP ausgeben kubectl get pod mypod \ -o jsonpath='{.status.podIP}' # Alle Images in allen Pods kubectl get pods -A \ -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.spec.containers[*].image}{"\n"}{end}' # Labels filtern kubectl get pods -l app=web,env=prod kubectl get pods -l 'app in (web,api)'
Labels & Annotations
# Labels setzen kubectl label pod mypod env=prod kubectl label pod mypod env- # entfernen # Annotation setzen kubectl annotate pod mypod \ owner=team-alpha # Alle Labels anzeigen kubectl get pods --show-labels

Apply & Manifest-Management
Apply vs. Create
# Deklarativ (empfohlen) kubectl apply -f manifest.yaml kubectl apply -f ./k8s/ # Verzeichnis kubectl apply -k ./overlays/prod # Kustomize # Imperativ (einmalig) kubectl create -f manifest.yaml kubectl replace -f manifest.yaml # Dry-Run (ohne Anwendung) kubectl apply -f manifest.yaml --dry-run=client kubectl apply -f manifest.yaml --dry-run=server
Manifest generieren
# Deployment-YAML ausgeben kubectl create deployment web \ --image=nginx \ --dry-run=client -o yaml > deploy.yaml # Patch anwenden kubectl patch deployment web \ -p '{"spec":{"replicas":5}}' # Live bearbeiten kubectl edit deployment web
Kustomize (integriert)
# kustomization.yaml Beispiel resources: - deployment.yaml - service.yaml images: - name: nginx newTag: "1.26" replicas: - name: web count: 5 # Preview kubectl kustomize ./overlays/prod
💡 --dry-run=server validiert gegen die echte API (Admission Webhooks!), --dry-run=client nur lokal. Für CI/CD immer server-seitig testen.

Nützliche Flags & Aliase
Kurzformen (Resource-Typen)
po → pods cm → configmaps svc → services sa → serviceaccounts deploy → deployments ep → endpoints sts → statefulsets ds → daemonsets pvc → persistentvolumeclaims cj → cronjobs ns → namespaces no → nodes sc → storageclasses
Nützliche Flags
-n NS # Namespace -A # alle Namespaces -o wide/yaml/json # Ausgabeformat -l key=val # Label-Selector --watch / -w # Live-Updates --dry-run=client # ohne Anwendung --force # erzwingen --grace-period=0 # sofortiges Löschen --record # Rollout-History (deprecated) --cascade=false # nicht abhängig löschen
Shell-Aliase (empfohlen)
# In ~/.bashrc oder ~/.zshrc alias k=kubectl alias kg='kubectl get' alias kd='kubectl describe' alias kl='kubectl logs' alias ke='kubectl exec -it' alias ka='kubectl apply -f' # kubectl-Autocomplete source <(kubectl completion bash) # Für alias k: complete -o default -F __start_kubectl k
💡 kubectx + kubens (von ahmetb): Komfortables Wechseln zwischen Contexts und Namespaces — kubens staging statt langen config-Befehlen.
