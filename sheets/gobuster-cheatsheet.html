Gobuster dir — Verzeichnis-Enum
Basis-Scan
gobuster dir \ -u http://target.tld \ -w /usr/share/wordlists/dirb/common.txt
Mit Datei-Endungen & Threads
gobuster dir \ -u http://target.tld \ -w directory-list-2.3-medium.txt \ -x php,html,txt,bak \ -t 50
Output, Status-Codes, Auth
gobuster dir \ -u http://target.tld \ -w raft-large-files.txt \ -s 200,204,301,302,307 \ -o results.txt \ -U admin -P password
HTTPS + benutzerdefinierter Header
gobuster dir \ -u https://target.tld \ -w common.txt \ -k # TLS-Fehler ignorieren -H "Authorization: Bearer TOKEN"
Proxy & erweitertes Timeout
gobuster dir \ -u http://target.tld \ -w common.txt \ --proxy http://127.0.0.1:8080 \ --timeout 10s
Flags: -e zeigt vollständige URLs · -r folgt Redirects · --no-error unterdrückt Fehler-Ausgaben
🌐
Gobuster dns — Subdomain-Enum
Subdomain-Brute-Force
gobuster dns \ -d target.tld \ -w subdomains-top1million-5000.txt
IPs anzeigen + eigener Resolver
gobuster dns \ -d target.tld \ -w subdomains-top1million-20000.txt \ -i \ -r 8.8.8.8
Wildcards manuell ignorieren
gobuster dns \ -d target.tld \ -w subdomains.txt \ --wildcard # erzwingt Scan trotz Wildcard
-i: gibt aufgelöste IP-Adressen aus · -r: nutzt eigenen DNS-Resolver · Wildcards werden standardmäßig erkannt und stoppen den Scan
🖧
Gobuster vhost — VHost-Enum
Virtual-Host-Brute-Force
gobuster vhost \ -u http://target.tld \ -w subdomains-top1million-5000.txt \ --append-domain
Falsch-Positive filtern
gobuster vhost \ -u http://10.10.10.10 \ -w subdomains.txt \ --append-domain \ --exclude-length 250
Mit Cookie & User-Agent
gobuster vhost \ -u http://target.tld \ -w subdomains.txt \ --append-domain \ -c "session=abc123" \ -a "Mozilla/5.0"
--append-domain: hängt die Basisdomain automatisch an jedes Wort an (z. B. admin → admin.target.tld). Ohne dieses Flag muss die Wortliste vollständige Hostnamen enthalten.
🪣
Gobuster s3 / gcs — Bucket-Enum
AWS S3-Buckets aufspüren
gobuster s3 \ -w bucket-names.txt
Google Cloud Storage Buckets
gobuster gcs \ -w bucket-names.txt
Mit Threads & Output
gobuster s3 \ -w bucket-names.txt \ -t 20 \ -o buckets.txt
Status-Bedeutung: 200 = Bucket existiert & öffentlich · 301 = existiert, Redirect · 403 = existiert, kein Zugriff · 404 = nicht gefunden

ffuf — Grundlagen
Basis-Fuzzing
ffuf \ -u http://target.tld/FUZZ \ -w /usr/share/seclists/Discovery/Web-Content/common.txt
Status-Codes & Threads
ffuf \ -u http://target.tld/FUZZ \ -w directory-list-2.3-medium.txt \ -mc 200,301,302,403 \ -t 40
Datei-Endungen fuzzen
ffuf \ -u http://target.tld/FUZZ \ -w common.txt \ -e .php,.html,.txt,.bak,.old
Zwei FUZZ-Schlüsselwörter (MSW)
ffuf \ -u http://target.tld/W1/W2 \ -w dirs.txt:W1 \ -w files.txt:W2 \ -mc 200
FUZZ ist das Standard-Schlüsselwort. Eigene Schlüsselwörter (W1, W2 …) erlauben gleichzeitiges Fuzzen mehrerer Positionen.
🔍
ffuf — Filter & Match
Match-Optionen (zeige nur …)
# Nur bestimmte HTTP-Statuscodes: ffuf -u URL/FUZZ -w wl.txt -mc 200,301 # Nur bestimmte Antwortgröße (Bytes): ffuf -u URL/FUZZ -w wl.txt -ms 1234 # Nur bestimmte Zeilenzahl: ffuf -u URL/FUZZ -w wl.txt -ml 42 # Nur bestimmte Wortanzahl: ffuf -u URL/FUZZ -w wl.txt -mw 10 # Regex-Match auf Antwort-Body: ffuf -u URL/FUZZ -w wl.txt -mr "Welcome"
Filter-Optionen (verstecke …)
# Bestimmte Status-Codes filtern: ffuf -u URL/FUZZ -w wl.txt -fc 404,403 # Bestimmte Antwortgröße filtern: ffuf -u URL/FUZZ -w wl.txt -fs 4242 # Bestimmte Zeilenzahl filtern: ffuf -u URL/FUZZ -w wl.txt -fl 10 # Bestimmte Wortanzahl filtern: ffuf -u URL/FUZZ -w wl.txt -fw 3 # Regex auf Body filtern: ffuf -u URL/FUZZ -w wl.txt -fr "Not Found"
Tipp: Erst ohne Filter laufen lassen, die häufigste Antwortgröße notieren, dann mit -fs SIZE filtern um False-Positives zu eliminieren.
📮
ffuf — POST & Header-Fuzzing
POST-Body fuzzen
ffuf \ -u http://target.tld/login \ -w passwords.txt \ -X POST \ -d "username=admin&password=FUZZ" \ -H "Content-Type: application/x-www-form-urlencoded" \ -fc 302
JSON-Body fuzzen
ffuf \ -u http://target.tld/api/login \ -w passwords.txt \ -X POST \ -d '{"user":"admin","pass":"FUZZ"}' \ -H "Content-Type: application/json" \ -mr "token"
Header-Wert fuzzen
ffuf \ -u http://target.tld/ \ -w headers.txt \ -H "X-Custom-Header: FUZZ" \ -mc 200
Mit Session-Cookie
ffuf \ -u http://target.tld/admin/FUZZ \ -w common.txt \ -b "session=abc123; role=user"
-b: Cookie-String direkt · -H: beliebige HTTP-Header · mehrere -H-Flags möglich
🏷️
ffuf — Subdomain & VHost
VHost-Fuzzing via Host-Header
ffuf \ -u http://target.tld/ \ -w subdomains-top1million-5000.txt \ -H "Host: FUZZ.target.tld" \ -fs 4242 # Standardgröße filtern
Subdomain-Enum (DNS-basiert)
ffuf \ -u http://FUZZ.target.tld/ \ -w subdomains-top1million-5000.txt \ -mc 200,301,302
IP mit Host-Header (kein DNS)
ffuf \ -u http://10.10.10.10/ \ -w subdomains.txt \ -H "Host: FUZZ.target.tld" \ -fc 400 \ -fs 1337
VHost vs. DNS: VHost-Fuzzing testet den Host-Header gegen eine IP — ideal wenn kein öffentliches DNS existiert. DNS-Fuzzing benötigt eine auflösbare Domain.
📊
ffuf — Ausgabe & Rate-Control
Output in verschiedenen Formaten
# JSON-Output: ffuf -u URL/FUZZ -w wl.txt \ -o results.json -of json # CSV-Output: ffuf -u URL/FUZZ -w wl.txt \ -o results.csv -of csv # HTML-Report: ffuf -u URL/FUZZ -w wl.txt \ -o report.html -of html
Rate-Limiting & Verzögerung
# Max. Requests pro Sekunde: ffuf -u URL/FUZZ -w wl.txt \ -rate 50 # Pause zwischen Requests (Sekunden): ffuf -u URL/FUZZ -w wl.txt \ -p 0.1 # Zufällige Pause (min-max): ffuf -u URL/FUZZ -w wl.txt \ -p 0.1-0.5
Proxy (z. B. Burp Suite)
ffuf \ -u http://target.tld/FUZZ \ -w common.txt \ -x http://127.0.0.1:8080 \ -replay-proxy http://127.0.0.1:8080
Interaktiver Modus & stilles Laufen
# Keine Fortschrittsanzeige (CI/Pipes): ffuf -u URL/FUZZ -w wl.txt -s # Maximale Laufzeit (Sekunden): ffuf -u URL/FUZZ -w wl.txt \ -maxtime 60 -maxtime-job 30
-replay-proxy: leitet nur Treffer (Matches) an den Proxy weiter — spart Burp-Traffic und erleichtert die manuelle Analyse.
