Grundsyntax
Allgemeine Syntax
hydra [Optionen] target service
Einzelner User, Passwort-Wordlist
hydra -l admin -P /usr/share/wordlists/rockyou.txt \ 192.168.1.10 ssh
User-Liste, Passwort-Liste
hydra -L users.txt -P pass.txt 10.0.0.1 ftp
Einzelnes Passwort testen
hydra -l root -p password123 target ssh
Wichtige Flags
-l user # einzelner Benutzername -L file # Benutzer-Wordlist -p pass # einzelnes Passwort -P file # Passwort-Wordlist -t 16 # parallele Tasks (default 16) -T 64 # globale parallele Tasks -o out.txt # Ergebnisse speichern -V # verbose (jeden Versuch zeigen) -v # verbose (weniger) -f # stop bei erstem Treffer (pro Host) -F # stop bei erstem Treffer (global)
Hinweis: Bei URI-Syntax service://target:port entfällt der separate Service-Parameter.

SSH & Telnet
SSH Standard-Port 22
hydra -l root -P wordlist.txt \ ssh://192.168.1.10
SSH abweichender Port
hydra -l admin -P pass.txt \ -s 2222 192.168.1.10 ssh
SSH mit URI-Syntax und Port
hydra -L users.txt -P pass.txt \ ssh://10.0.0.1:2222
Telnet
hydra -l admin -P pass.txt \ telnet://192.168.1.1
SSH mit 4 Tasks (weniger aggressiv)
hydra -l root -P rockyou.txt \ -t 4 ssh://192.168.1.10
Tipp: SSH-Server limitieren oft Verbindungen. -t 4 oder weniger verhindert Aussperrung.
📁
FTP & SMB
FTP Standard-Port 21
hydra -l ftpuser -P pass.txt \ ftp://192.168.1.10
FTP mit Passwort-Liste und Verbose
hydra -L users.txt -P pass.txt \ -V ftp://10.0.0.5
SMB (Windows File Shares)
hydra -l administrator -P pass.txt \ smb://192.168.1.20
SMB mit User-Liste
hydra -L users.txt -P pass.txt \ -t 1 smb://192.168.1.20
FTP FTPS (TLS)
hydra -l user -P pass.txt \ ftps://192.168.1.10
SMB-Hinweis: SMBv1 erfordert -t 1. Bei Domain-Konten Format DOMAIN\\user in der Userliste verwenden.
🌐
HTTP-Form
HTTP POST-Formular (häufigstes Szenario)
hydra -l admin -P pass.txt \ 192.168.1.10 \ http-post-form "/login:user=^USER^&pass=^PASS^:Invalid"
HTTP POST-Form — Struktur des dritten Parameters
# Format: "URL:POST-Body:Fehlerstring" # ^USER^ und ^PASS^ sind Platzhalter http-post-form "/admin/login.php:\ username=^USER^&password=^PASS^:\ F=Login failed"
HTTP GET-Formular
hydra -l admin -P pass.txt \ 192.168.1.10 \ http-get-form "/login?u=^USER^&p=^PASS^:Error"
HTTPS POST-Formular
hydra -l admin -P pass.txt \ -s 443 192.168.1.10 \ https-post-form "/login:u=^USER^&p=^PASS^:denied"
Mit Cookie / Session-Token
hydra -l admin -P pass.txt \ 192.168.1.10 http-post-form \ "/login:u=^USER^&p=^PASS^:F=fail:H=Cookie\: sess=abc"
Erfolgsbedingung statt Fehlerbedingung
# S= für "success string" (statt F= für failure) http-post-form "/login:u=^USER^&p=^PASS^:S=Dashboard"
Tipp: Browser-DevTools (Network-Tab) zum Ermitteln des exakten POST-Body und der Fehlerseiten-Textmuster verwenden.

RDP & VNC
RDP (Windows Remote Desktop, Port 3389)
hydra -l administrator -P pass.txt \ rdp://192.168.1.100
RDP mit abweichendem Port
hydra -l admin -P pass.txt \ rdp://192.168.1.100:3390
RDP mit User-Liste, stop-on-success
hydra -L users.txt -P pass.txt \ -f -t 4 rdp://192.168.1.100
VNC (Port 5900, nur Passwort)
hydra -P pass.txt vnc://192.168.1.50
VNC mit abweichendem Port
hydra -P pass.txt -s 5901 \ 192.168.1.50 vnc
RDP-Hinweis: Windows sperrt Konten nach Fehlversuchen. -t 1 und Wartezeit -W 30 empfohlen.

hydra -l root -P pass.txt \ mysql://192.168.1.10

hydra -l dbadmin -P pass.txt \ mysql://192.168.1.10:3307
Microsoft SQL Server (Port 1433)

PostgreSQL (Port 5432)
hydra -l postgres -P pass.txt \ pgsql://192.168.1.10

Wortlisten & User-Listen
User- und Passwort-Listen (Kombination)
hydra -L users.txt -P pass.txt \ ssh://192.168.1.10
Kombinationsdatei (-C): user:pass pro Zeile
# Format der Combo-Datei: user:password hydra -C combo.txt ssh://192.168.1.10
Colon-Separator in Combo-Datei
# combo.txt Inhalt: admin:password123 root:toor user:letmein
Bekannte Wordlist-Pfade (Kali/ParrotOS)
/usr/share/wordlists/rockyou.txt /usr/share/wordlists/fasttrack.txt /usr/share/seclists/Passwords/ /usr/share/seclists/Usernames/
Einzelner User, einzelnes Passwort (Test)
hydra -l admin -p admin \ ftp://192.168.1.10
SecLists: Das Paket seclists (apt install seclists) enthält spezialisierte Listen für Web-Logins, Default-Credentials, etc.
⚙️
Performance & Rate Control
Parallele Tasks pro Ziel-Host
hydra -t 16 # default: 16 Tasks pro Host hydra -t 4 # für SSH/RDP schonender hydra -t 1 # für SMB v1 notwendig
Globale parallele Tasks (Multi-Target)
hydra -T 64 # default: 64 simultane Verbindungen
Wartezeit zwischen Verbindungen
hydra -W 3 # 3 Sekunden warten hydra -W 0 # keine Wartezeit (default)
Cooldown nach Fehler (Exit-Delay)
hydra -c 5 # 5 Sekunden nach Fehlversuch
Verbindungs-Timeout
hydra -w 30 # Max. Wartezeit auf Antwort (Sek.)
Unterbrochenen Angriff fortsetzen (-R)
# Hydra speichert Fortschritt in hydra.restore hydra -R # Resume letzten Session hydra -I # restore-Datei ignorieren, neu starten
Tipp: CTRL+C unterbricht sauber und schreibt hydra.restore. Mit -R nahtlos fortsetzen.
📄
Ausgabe & Ergebnisse
Ergebnisse in Datei schreiben
hydra -l admin -P pass.txt \ -o results.txt ssh://192.168.1.10
Ausgabeformat wählen (-b)
hydra ... -b text # Plain Text (default) hydra ... -b jsonv1 # JSON v1 hydra ... -b json # JSON
Fund-Format in der Ausgabe
# Erfolgreiche Credentials erscheinen als: [PORT][SERVICE] host: TARGET login: USER password: PASS
Stop bei erstem Fund
hydra -f # stop nach Fund pro Host hydra -F # stop nach erstem Fund überhaupt
Vollständige Beispiel-Ausgabe
[22][ssh] host: 192.168.1.10 login: root password: toor 1 of 1 target successfully completed, 1 valid password found
Alle gültigen Logins ausgeben (-u)
# -u: User-Liste iteriert äußere Schleife # (statt Passwort-Liste) — bei großen Listen hydra -u -L users.txt -P pass.txt \ ssh://192.168.1.10
JSON-Ausgabe: -b jsonv1 -o results.json eignet sich für Weiterverarbeitung mit jq in Automatisierungspipelines.
